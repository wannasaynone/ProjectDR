## 實作要點

### 架構決策：單一 CommissionManager 支援多角色

**原題目要求**：B5 提到 CommissionManager / CraftManager / GatheringManager 三個 Manager。

**實作選擇**：單一 `CommissionManager` 管理所有角色的 slots，依 `character_id` 分群。

**原因**：
1. **資料結構完全同構** — GDD v1.1 § 3.2「單物品配方」明確規定三類委託皆為「單一輸入 → 單一產出 + 倒數時間」，且所有配方都放在同一份 `commission-recipes-config.json`，只靠 `character_id` 區分。三個 Manager 會導致三套幾乎相同的程式碼。
2. **紅點系統 L1 統一查詢** — 紅點系統需查詢「哪個角色有完成委託」。單一 Manager 可提供統一 API（`GetManagedCharacterIds` + `GetSlots(characterId)`），否則 B7 紅點系統需遍歷三個 Manager 的介面。
3. **事件統一** — 所有委託共用同一組事件（`CommissionStartedEvent` 等），訂閱者不需分三類。
4. **擴充性** — 未來增加委託型角色只需加 config 條目，無需新增程式類別。

### FarmManager 處置策略：三路徑取捨

**背景**：`FarmManager` 已實作農女耕種邏輯（42 個通過測試），UI 層 `FarmAreaView` 深度耦合 FarmManager API。

三條可能路徑：

| 路徑 | 做法 | 優點 | 缺點 |
|------|------|------|------|
| **A. 保留共存** | CommissionManager 白名單過濾農女，FarmManager 繼續服務農女 | 不破壞既有測試與 UI、B5 範疇可控 | 農女走兩條路徑，最終需要統一 |
| **B. 合併** | 把 FarmManager 改寫為 CommissionManager 的一部分，刪除 FarmManager | 邏輯唯一 | B5 需連帶重寫 FarmAreaView（超出範疇，屬於 B11） |
| **C. Adapter** | FarmManager 改為 CommissionManager 的 adapter | 可保留既有 UI 與測試、邏輯唯一 | adapter 層增加維護成本，且語意落差（FarmPlot vs CommissionSlot）需要大量 marshalling |

**選擇路徑 A（保留共存）**，原因：
- B5 的工作範圍是「純邏輯管理器層」，UI 統一整合是 B11 的工作（提煉介面 UI 格子式工作台）
- B11 將重寫 FarmAreaView 為統一的「提煉介面」View，到時再把農女切換到 CommissionManager，刪除 FarmManager 與 FarmAreaView，此為自然演進路徑
- 保留 FarmManager 的 42 個測試為既有資產，避免 B5 一次性破壞過多測試
- 透過 `CommissionManager` 建構子的 `allowedCharacterIds` 白名單，本 Sprint 僅啟用 Witch / Guard，避免與 FarmManager 雙重管理農女

**演進路徑**：B11 完成後，將 `allowedCharacterIds` 改為包含 FarmGirl，同時刪除 `FarmManager`、`SeedData`、`FarmPlot`、`FarmAreaView`、`FarmPlotUI.prefab`、`SeedItemButton.prefab`、`FarmManagerTests.cs`，並將 `VillageEntryPoint` 中的 `_farmManager` 相關組裝移除。

### 背包/倉庫整合策略：先背包後倉庫

**輸入扣除**（StartCommission）：
1. 檢查 `backpack.GetItemCount + storage.GetItemCount >= 所需量`，否則 InsufficientInput
2. 先從背包扣至背包空，剩餘從倉庫扣
3. 若倉庫扣失敗（極罕見的 race condition），回滾背包已扣部分

**產出入庫**（ClaimCommission）：
1. 先嘗試 `backpack.AddItem`（回傳實際加入數量）
2. 剩餘用 `storage.TryAddItem`
3. 若合計 < 應領取量，**回滾兩邊已加入的部分**，slot 保留 Completed 狀態，讓玩家清出空間後再 Claim
4. 回傳 `NoSpaceForOutput` 錯誤

此策略比照 `GiftManager` 的既有先背包後倉庫模式。

### 事件發布細節

- `CommissionStartedEvent`：扣完輸入後發布一次
- `CommissionCompletedEvent`：在 Tick 或 Claim 時偵測到 slot 從 InProgress 跨越到 Completed 邊界時發布**一次**（透過 `slot.State` 狀態機去重）
- `CommissionClaimedEvent`：物品成功入倉儲、slot 重設為 Idle 後發布
- `CommissionTickEvent`：整秒剩餘秒數變化時發布（透過 `slot.LastPublishedRemainingSeconds` 去重，避免每幀事件風暴）
  - 額外：在轉 Completed 時會多發布一次 `RemainingSeconds=0` 的 Tick，讓倒數 UI 顯示「00:00」再切換為「可領取」

### 主線任務 commission_count 訊號連線

GDD `main-quest-system.md` v1.1 定義 `commission_count` 為 T2/T3 的完成條件型。實作：

```csharp
// VillageEntryPoint
EventBus.Subscribe<CommissionClaimedEvent>(OnCommissionClaimedForMainQuest);

private void OnCommissionClaimedForMainQuest(CommissionClaimedEvent e)
{
    _mainQuestManager.NotifyCompletionSignal(
        MainQuestCompletionTypes.CommissionCount, e.CharacterId);
}
```

**注意**：目前 `main-quest-config.json` 的 T2/T3 `completion_condition_value` 為 placeholder（`choice1_character|1`、`choice2_character|1`，內含 `|1` 計數標記與 placeholder 角色名）。**本次實作僅傳送 `characterId` 作為 signalValue**，與 placeholder 不匹配是預期行為。

B7（紅點 4 層）與 C1（端到端整合）階段需：
1. 將 `main-quest-config.json` 的 T2/T3 `completion_condition_value` 改為實際選定的角色 ID（根據節點 1/2 的 VN 選項決定）
2. 或擴充 `MainQuestManager.NotifyCompletionSignal` 的匹配邏輯，支援 `characterId|count` 累計型比對

此待釐清事項已記錄於 project-status.md。

### 可訂閱的事件

| 事件 | 用途 |
|------|------|
| `CommissionStartedEvent` | 開始委託 → B11 提煉介面關閉、B12 工作中狀態顯示倒數 |
| `CommissionCompletedEvent` | slot 完成 → B7 紅點 L1 觸發、B12 工作中狀態切為「可領取」 |
| `CommissionClaimedEvent` | 領取完成 → B11 提煉介面重置（回 Idle） |
| `CommissionTickEvent` | 整秒倒數變化 → B12 工作中狀態更新 `mm:ss` 文字 |

### DTO 欄位可查詢清單

**`CommissionSlotInfo`（struct）**：
- `CharacterId`, `SlotIndex`, `State`（Idle/InProgress/Completed）
- `RecipeId`, `OutputItemId`, `OutputQuantity`
- `StartedTimestampUtc`, `DurationSeconds`, `RemainingSeconds`（整秒）

**`CommissionRecipeInfo`（class）**：
- `RecipeId`, `CharacterId`
- `InputItemId`, `InputQuantity`（空手委託時兩者為空/0）
- `OutputItemId`, `OutputQuantity`
- `DurationSeconds`, `WorkbenchSlotIndexMax`, `Description`
- `IsEmptyHanded`（判斷空手委託輔助屬性）

### 提煉介面 UI（B11）建議流程

1. 進入角色提煉介面：
   - `recipes = config.GetRecipesByCharacter(characterId)` 取得該角色所有可做配方
   - `slots = manager.GetSlots(characterId)` 取得所有 slot 狀態
2. 在每個 Idle slot 顯示：
   - 可選配方列表（可依 `recipe.IsEmptyHanded` 判斷顯示「空手委託」樣式）
   - 玩家背包+倉庫擁有該 `recipe.InputItemId` 的總數（從 BackpackManager/StorageManager 查）
3. 玩家點擊「開始」：呼叫 `StartCommission(characterId, recipeId, slotIndex)`
4. 工作中 slot 顯示：
   - 倒數 `slot.RemainingSeconds`（格式 `mm:ss`，訂閱 `CommissionTickEvent` 更新）
5. 完成 slot 顯示：
   - 產出預覽（`slot.OutputItemId × slot.OutputQuantity`）+ 領取按鈕
   - 點擊 → 呼叫 `ClaimCommission(characterId, slotIndex)`
   - 若回傳 `NoSpaceForOutput`，顯示「背包倉庫已滿」提示

### 工作中狀態 UI（B12）建議流程

1. CharacterInteractionView 進入時：
   - 檢查 `manager.GetSlots(characterId)` 是否有 InProgress 的 slot
   - 若有，切換為工作中狀態（隱藏功能選單，顯示倒數）
2. 訂閱 `CommissionTickEvent`：篩選 `e.CharacterId == 當前 characterId` 時更新文字
3. 訂閱 `CommissionCompletedEvent`：切換為「可領取」狀態（Hub 紅點 L1 亮起）
