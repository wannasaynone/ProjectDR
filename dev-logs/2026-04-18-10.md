# 對話按鈕紅點與角色發問回應流程修正

> **日期**：2026-04-18
> **階段**：Internal Test（實機回饋修正）
> **類型**：Bug Fix

## 背景

製作人實機測試 Sprint 5 完成後的對話功能，回報兩個問題：

1. 角色主動發話（L2 紅點亮起）時，**村莊 Hub** 的角色按鈕紅點正確顯示，但**進入角色頁面**後「對話」按鈕沒有紅點。
2. 角色頁面選擇「對話」後，玩家選擇答案，回應（response）被顯示在 `CharacterQuestionsView` overlay 內，畫面看起來像 CG；製作人預期應該關閉 overlay、返回角色互動主畫面，由主畫面的對話區顯示回應。

## 根因分析

### Bug 1：對話按鈕紅點不顯示

`CharacterInteractionView.ApplyButtonRedDots()` 的邏輯正確（`IsLayerActive` 查 L2 → 找 AreaButton 子物件 "RedDot" → SetActive）。
根因在 `AreaButton.prefab` 的 `RedDot` 子物件資產：

- Color = 白色 (1,1,1,1)
- Size = 100×100（置中）
- Anchor/Pivot = (0.5, 0.5)
- Sprite = null

結果：紅點 SetActive(true) 後是一塊**白色大方塊覆蓋按鈕中央**，視覺上不像紅點、甚至與按鈕底色相近看不出來。

### Bug 2：回應不該留在 overlay

`CharacterQuestionsView.OnOptionClicked` 原流程：

1. `_manager.SubmitAnswer(...)`
2. `_typewriter.Play(response)` 在 overlay 內的 `_dialogueText` 播回應
3. 回應播完後等玩家按關閉按鈕才離開

overlay 本身是 w=1600 h=1800 中央蓋滿畫面的 Panel，對玩家而言就是一張「CG 對話畫面」。規格應該是返回角色互動主畫面（含立繪與主對話區）顯示回應。

## 修正

### 1. `AreaButton.prefab` RedDot 視覺調整（Unity MCP）

透過 `mcp__UnityMCP__manage_components` 於 PrefabStage 內直接改 RedDot 子物件：

- `UnityEngine.UI.Image`：color = (0.9, 0.2, 0.2, 1.0)、`raycastTarget = false`
- `UnityEngine.RectTransform`：anchorMin/Max = (1, 1)、pivot = (1, 1)、sizeDelta = (48, 48)、anchoredPosition = (-16, -16)

結果：活化時在按鈕右上角顯示紅色 48×48 方塊（與 VillageHubView 程式化建立的 Hub 紅點視覺一致）。

### 2. `CharacterInteractionView` 新增 `PlayDialogue(string[] lines)` public API

- 行為：`CloseOverlay` → `RefreshAffinityDisplay` → `SetMenuVisible(false)` → 啟用全螢幕點擊 → 訂閱 `DialogueCompletedEvent` → `StartDialogue` + 打字機首行
- `OnGiftSuccess` 重構為先組裝 ThankYouLines，再呼叫 `PlayDialogue(lines)`（共用同一段邏輯）

### 3. `CharacterQuestionsView` 選答案改為委外播放（a 路徑）

- 新增 `_responseAction : Action<string>` 欄位 + `SetResponseAction(Action<string>)` 注入 API
- `OnOptionClicked`：`SubmitAnswer` → `ClearOptions` → `_responseAction.Invoke(response)`
- 移除 `_responseShown` 與 `OnResponseTypewriterComplete` 的 overlay 內 response 打字機播放邏輯
- 未設定 responseAction 時退回僅 `_returnAction.Invoke()`（保留關閉 overlay 的基本行為）

### 3b. `PlayerQuestionsView` 選題改為委外播放（b 路徑，魔女等無 L2 紅點時走此路徑）

製作人回報「魔女的沒修到」：魔女在沒有 L2 紅點時走 PlayerQuestionsView（b 路徑），其 `ShowAnswerPanel` 原本也在 overlay 內播回答，同樣看起來像 CG。

- 同樣新增 `_responseAction : Action<string>` 欄位 + `SetResponseAction(Action<string>)` 注入 API
- `OnQuestionSelected` / `OnIdleChatSelected`：扣體力 / 啟 CD / MarkSeen 之後，若 `_responseAction != null` 則 `Invoke(response)`；否則退回舊 ShowAnswerPanel 行為（保留向下相容，便於舊測試或未接 Presenter 時可獨立跑）

### 4. `VillageEntryPoint` 註冊 responseAction（a 與 b 雙路徑）

在 `RegisterFunctionPrefabs` 中為 `FunctionIds.CharacterQuestion` 與 `FunctionIds.Dialogue` 雙路徑初始化回呼新增：

```csharp
cqView.SetResponseAction(response =>
    interactionView.PlayDialogue(new string[] { response }));

questionsView.SetResponseAction(response =>
    interactionView.PlayDialogue(new string[] { response }));
```

`PlayDialogue` 內部會先呼叫 `CloseOverlay`，因此一個 callback 即同時達成「關閉 overlay」與「在主畫面播放 response」兩件事。

## 預期驗證流程（製作人實機）

1. 進入遊戲，等 60s 倒數 → 某角色 Hub 紅點亮（既有行為）
2. 點該角色 → 進入互動畫面播招呼 → 打完選單出現時「對話」按鈕右上角**亮紅色小方塊**
3. 點對話 → overlay 開啟播 Prompt + 4 選項
4. 選任一選項 → overlay 立刻關閉，回到角色互動主畫面，**立繪依舊在**，主對話區以打字機播放對應 response
5. 回應播完 → 自然回到功能選單

## 追加功能：pending 節點期間 VCW 對話隱藏 / 女角立刻紅點

### 需求（製作人三次澄清後的最終規則）

前期解鎖流程進行期間（node_1 與 node_2 都尚未播畢），即整個「認識角色」階段：

- **村長夫人的「對話」按鈕隱藏**：此期間每次進入 VCW 都有可能自動播放節點劇情，避免玩家誤觸一般發問路徑
- **其他女性角色（農女/魔女）對話按鈕保留且立刻亮 L2 紅點**：鼓勵玩家先與新解鎖角色互動，不需等 60s 倒數

**第一版**：鎖非 VCW 角色（方向相反，已作廢）
**第二版**：鎖 VCW，以 `GetPendingMainQuestNodeId() != null` 為條件（T1/T3 完成→節點未播間隙才有效，其他時點 VCW 對話還是可按）
**第三版**（本版）：鎖 VCW，以「node_1 與 node_2 皆已播放」為流程結束判定

### 判定條件

VCW 的對話按鈕隱藏判定（`SetDialogueSuppressionProvider`）：

- characterId == VillageChiefWife
- `!HasPlayedAllUnlockNodes()`（`_playedMainQuestNodes` 未同時包含 node_1 與 node_2）

兩條件同時成立 → 隱藏 VCW 的「對話」按鈕。

女角立即紅點：`CharacterUnlockedEvent` → 若 characterId 為 農女 或 魔女 → `RedDotManager.SetCharacterQuestionFlag(id, true)`。

### 實作

1. `CharacterInteractionView` 新增：
   - `SetDialogueSuppressionProvider(Func<bool>)` 注入判定函式
   - `_visibleFunctionIds` 儲存本次 RefreshMenu 實際建立的按鈕順序，讓 `ApplyButtonRedDots` 依此索引正確對應功能 ID（原先依 `_characterData.FunctionIds` 索引，按鈕被跳過時會錯位）
   - `RefreshMenu` 在建立按鈕時跳過 `FunctionIds.Dialogue`（若 provider 回 true）
   - 訂閱 `MainQuestCompletedEvent` / `NodeDialogueCompletedEvent` → 活性期間自動 `RefreshMenu`
2. `VillageEntryPoint.InitializeCharacterView` 注入（第三版，現行）：
   ```csharp
   string capturedCharacterId = characterId;
   interactionView.SetDialogueSuppressionProvider(() =>
       capturedCharacterId == CharacterIds.VillageChiefWife
       && !HasPlayedAllUnlockNodes());
   ```
3. `VillageEntryPoint` 新增 `HasPlayedAllUnlockNodes()`：回傳 `_playedMainQuestNodes` 同時含 `NodeIdNode1` 與 `NodeIdNode2`
4. `VillageEntryPoint` 新增 `OnCharacterUnlockedForDialogueRedDot`：訂閱 `CharacterUnlockedEvent`，若解鎖角色為 農女/魔女，呼叫 `_redDotManager.SetCharacterQuestionFlag(id, true)` 立刻亮 L2 紅點

### 紅點索引錯位修正（Destroy 延遲導致）

症狀：農女解鎖 → Hub 紅點正確 → 進入互動畫面 → 播放對話正確 → **菜單顯示但對話按鈕沒紅點**，但點擊仍走 a 路徑（角色發問，代表 L2 其實是 active 的）。

根因：`RefreshMenu` 先 `Destroy` 舊按鈕再 `Instantiate` 新按鈕，Unity 的 `Destroy` 為**延遲銷毀**（當幀結束才真正移除 GameObject）。在同一 frame 內接著呼叫 `ApplyButtonRedDots` 時，`_menuContainer.childCount` 同時包含**舊按鈕（即將 destroy）+ 新按鈕**，索引 0~(oldCount-1) 對應舊按鈕、(oldCount)~(oldCount+newCount-1) 才是新按鈕。ApplyButtonRedDots 卻以 `_visibleFunctionIds[i]` 作索引，等於把紅點設到舊按鈕上（這些即將消失），新按鈕完全沒設到。

修正：`CharacterInteractionView` 新增 `_visibleButtons : List<Button>`，`RefreshMenu` 中每 Instantiate 一個 button 就 `_visibleButtons.Add(button)`，`ApplyButtonRedDots` 改為 iterate `_visibleButtons`（同時用 Unity 的 `btn == null` overload 跳過已銷毀的實例）。完全避開 `_menuContainer.childCount` 迭代，確保紅點只套用到本輪存活的按鈕。

### 預期行為驗證

- 節點 0 選擇 A 女角 → A 解鎖瞬間：L2 紅點已亮在 A 的對話按鈕（由 `OnCharacterUnlockedForDialogueRedDot` 觸發）

- B 解鎖（node_1 的選擇 2）→ B 立刻亮 L2 紅點（同 A 邏輯）

## 追加功能：a 路徑角色發問改為 inline VN 式流程

### 需求

製作人指定正確流程：
```
按下對話 → 右下選單關閉 → 右上顯示對話（prompt）→ 右下顯示回答（4 選項）
→ 點擊回答 → 回答選單關閉 → 右上顯示對話（response）→ 右下選單回復
```
`CharacterQuestionsView` overlay 方式被視為 CG 感太重，且不符合 UI 分工。

### 實作

`CharacterInteractionView` 新增：

- `SetCharacterQuestionDependencies(CharacterQuestionsManager, CharacterQuestionCountdownManager)` — 注入相依
- `StartCharacterQuestionInline()` — 清 L2 + ClearReady + PickNextQuestion + 隱藏選單 + 打字機播 prompt + 重啟 60s 倒數
- `OnCharacterQuestionPromptComplete()` — 打字機完成 → 呼叫 `ShowCharacterQuestionChoices`
- `ShowCharacterQuestionChoices(CharacterQuestionInfo)` — 於 `_choiceContainer` 生成 4 個選項按鈕（UI 只顯示 Text，不顯示 +N 數值）
- `OnCharacterQuestionOptionSelected(personality, response)` — 提交答案 + 清選項 + `PlayDialogue(new[]{response})`（完成後 `OnDialogueCompleted` 自然 `SetMenuVisible(true) + RefreshMenu`，右下選單回復）

題目池耗盡 fallback：`PlayDialogue(new[]{"（目前沒有更多問題。）"})`。

`OnFunctionClicked(Dialogue)` 分支改寫：
```csharp
if (hasL2 && _characterQuestionsManager != null)
    StartCharacterQuestionInline();
else if (_functionPrefabs.ContainsKey(FunctionIds.Dialogue))
    OpenOverlay(FunctionIds.Dialogue);   // b 路徑仍走 PlayerQuestionsView
```

### Prefab 調整

`CharacterInteractionView.prefab` 的 `ChoiceContainer` RectTransform 改為與 `MenuPanel` 對齊：
- anchorMin (0.4, 0.0)、anchorMax (1.0, 0.4)、anchoredPosition (-5, 35)、sizeDelta (-30, -90)

選項現在出現於原選單位置（右下），符合「右下顯示回答」規格。

### HCG 搶先 UI 修正（延後 SubmitAnswer）

製作人回報 inline 排版正確，但點選項後 response 沒播到，畫面直接變 CG 對話。根因：`OnCharacterQuestionOptionSelected` 立刻呼叫 `CharacterQuestionsManager.SubmitAnswer` → 進入 `AffinityManager.AddAffinity` → 跨越門檻時 publish `AffinityThresholdReachedEvent` → `CGUnlockManager` publish `CGUnlockedEvent` → `VillageEntryPoint.OnCGUnlocked` **同步**執行 `_uiContainer.SetActive(false) + HCGDialogueSetup.PlayCGScene`。整條鍊在 `PlayDialogue(response)` 還沒開始播之前完成 UI 蓋掉 → 玩家看不到 response，直接看到 HCG。

修正：延後 `SubmitAnswer` 到 response 播完後才執行。

- `CharacterInteractionView` 新增 `_pendingCharacterQuestionSubmit` 旗標 + `_pendingSubmitQuestionId` / `_pendingSubmitPersonality`
- `OnCharacterQuestionOptionSelected` 改為僅記錄參數 + 立即 `PlayDialogue(response)`，**不**立刻 SubmitAnswer
- `SubmitPendingCharacterQuestionAnswerIfAny()` 新方法
- `OnDialogueCompleted` 先呼叫 `SubmitPendingCharacterQuestionAnswerIfAny` 再 `SetMenuVisible(true) + RefreshMenu`，如此 HCG 若被觸發會接在「response 播完 + 選單已恢復」之後（HCG 結束時 UI 會還原到有選單的狀態）
- `OnHide` 保險呼叫一次，避免玩家 response 沒播完就返回 Hub 導致答案未計入

修正後流程：
1. 點選項 → ClearChoiceContainer（右下關閉）
2. PlayDialogue 啟動 → 主對話區播 response（右上顯示）
3. 玩家推進至 response 結束 → OnDialogueCompleted
4. SubmitPendingCharacterQuestionAnswer → AddAffinity（可能觸發 HCG 非同步開始）
5. SetMenuVisible(true) + RefreshMenu（選單設定為可見）
6. HCG 若有觸發：_uiContainer 暫時 SetActive(false) → HCG 播完 → SetActive(true) → 選單自然重新顯示
7. HCG 無觸發：選單維持可見

### CharacterQuestionsView 檔案狀態

保留但已無呼叫路徑觸發（`OnFunctionClicked` 不再 `OpenOverlay(CharacterQuestion)`）。VillageEntryPoint 仍有註冊但不會被啟動。後續清理可移除。
