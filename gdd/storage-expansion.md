# 倉庫擴建系統（Storage Expansion）

> **版本**: v1.1

## 系統目的

倉庫擴建系統承接 `core-definition.md` v3.1 的核心樂趣 **#2 經營成就感**——玩家探索帶回的物資具象化為「容量成長」的可見進度，每一次擴建都是探索積累的證明。

本系統同時對應製作人 2026-04-17 第四輪 A1 決策的**村長夫人定位**：她是劇情型 + 功能型角色（倉庫 + 主線任務 + 擴建），不走常規委託。倉庫擴建作為村長夫人的主要交互循環之一，維持她的接觸頻率（而非委託型的高頻接觸）。

設計依據：
- 製作人 2026-04-17 第五輪 DA 決策：**擴建容量**（選項 A）
- 製作人 2026-04-17 第六輪 EA-1/2/3 決策：**格子制 + 物品堆疊**、**初始 100 格**、**每次 +50 格**

## 一、倉庫規格

### 1.1 資料結構（格子制 + 物品堆疊）

製作人 2026-04-17 第六輪 EA-1 決策：倉庫採用**格子制 + 物品堆疊（有堆疊上限）**，**與背包規格相同**。

```
StorageSlot {
    ItemId          // 格子內物品 ID（空格為 null/empty）
    Quantity        // 當前堆疊數量
    MaxStack        // 堆疊上限（由 ItemData 定義，與背包一致）
}

Storage {
    Capacity        // 格子總數（初始 100，每次擴建 +50）
    Slots[]         // 格子陣列（長度 = Capacity）
}
```

### 1.2 容量規格

| 項目 | 規格 |
|---|---|
| **初始容量** | **100 格**（EA-2） |
| **每次擴建增量** | **+50 格**（EA-3） |
| **擴建次數上限** | TBD（預設無上限，可無限擴建） |
| **堆疊上限** | 與背包規格相同（每物品各自定義） |

### 1.3 倉庫與背包的關係

倉庫容量規格**與背包規格完全一致**（格子制 + 堆疊）：

| 比較 | 背包（既有 BackpackManager） | 倉庫（新擴建 StorageManager） |
|---|---|---|
| 結構 | 格子制 + 堆疊 | 格子制 + 堆疊 |
| 容量 | 背包容量（已有定義） | 初始 100 + 每次擴建 50 |
| 物品堆疊上限 | 每物品 MaxStack | 與背包同（共用 ItemData） |
| 轉移 | 既有 `StorageTransferManager` 支援雙向轉移 | 同左，規格一致後更簡化 |

設計優勢：規格統一使 UI、轉移邏輯、溢出處理三者可共用同一套設計語言。

### 1.4 既有 StorageManager 的改造範圍

| 項目 | 現況 | 變更後 |
|---|---|---|
| `StorageManager` 容量 | 無上限 | **引入 Capacity 參數 + 擴建 API** |
| 物品儲存結構 | 字典（itemId → count） | **改為 `StorageSlot[]`（格子制 + 堆疊）** |
| `StorageAreaView` UI | 顯示庫存清單 | **需顯示格子與堆疊 + 擴建入口** |
| 物品放入邏輯 | 無檢查 | **檢查容量/堆疊，超過時依 TBD 策略處理** |
| 物品轉移（`StorageTransferManager`） | 字典對字典 | **格子對格子，邏輯與背包內部堆疊一致** |

**注意**：此改造屬於中量級程式重構，屬於 Phase D5 之後的實作工作，本 GDD 僅定義設計規格。

## 二、擴建流程

### 2.1 擴建入口

TBD：擴建入口尚未最終決定，可能方案：

| 方案 | 描述 |
|---|---|
| **方案 A：倉庫介面內的擴建按鈕** | 玩家進入村長夫人 → 點 `[倉庫]` → 在倉庫 View 內看到「擴建」按鈕 |
| **方案 B：新增 `[擴建]` 功能按鈕** | 村長夫人功能選單增為 `[倉庫][擴建][任務][對話][送禮]`（五項） |
| **方案 C：整合至倉庫 View 外層** | 倉庫 View 下方或側邊顯示「目前容量 X/Y + 擴建」區塊 |

**工作室建議方案 A**（倉庫介面內擴建按鈕）—— 理由：
- 保持村長夫人功能選單為 4 項（與其他三位角色選單項數一致，UX 一致性）
- 擴建是倉庫的屬性操作，在倉庫介面內操作語意一致
- 降低 UI 複雜度

製作人最終決定待確認（HB-1）。本 GDD 暫以方案 A 為基準描述流程。

### 2.2 擴建完整流程（基準方案 A）

```
Hub 畫面 → 點擊村長夫人
  ↓
CharacterInteractionView — 第一狀態
  對話播放 → 功能選單 [倉庫][送禮][CG 圖鑑][對話]
  ↓ 點擊 [倉庫]
StorageAreaView（Overlay 顯示於 interact view 上）
  - 顯示當前格子狀態（X/Y 格使用中）
  - 顯示「擴建」按鈕
  ↓ 點擊「擴建」
擴建確認介面
  - 顯示：當前容量 → 擴建後容量（+50 格）
  - 顯示：所需物資（TBD，後續 session 設定）
  - 顯示：擴建等待時間（TBD，遵循現實時間等待原則）
  - [確認擴建] / [取消] 按鈕
  ↓ 玩家確認
系統扣除物資（先背包後倉庫優先級，與既有邏輯一致）
  ↓
切回 interact view，村長夫人播放擴建敘事對話
  範例：「這就著手安排。忙完我會告訴你。」
  功能選單變為「擴建中 mm:ss」倒數顯示（類同委託的工作中狀態）
  ↓ 玩家離開，可做別的事
【倒數進行中】
  ↓ 倒數完成
Hub 畫面村長夫人頭像亮紅點（L1 擴建完成層）
  ↓ 玩家點擊村長夫人
CharacterInteractionView — 擴建完成狀態
  村長夫人播放完成對話
  範例：「倉庫已經整理好了，多了 50 格給你放東西。」
  ↓ 確認
倉庫容量更新（+50 格），擴建完成
  ↓
回到第一狀態
```

### 2.3 擴建狀態機

| 狀態 | 說明 | 可執行操作 |
|---|---|---|
| **Idle（可擴建）** | 無擴建進行中 | 檢查物資 → 確認擴建 |
| **InProgress（擴建中）** | 倒數進行中 | 可進入村長夫人畫面（工作中狀態） |
| **Completed（可領取）** | 倒數完成，紅點亮 | 進入村長夫人播放完成對話、容量生效 |

### 2.4 擴建等待時間

遵循委託系統的現實時間等待原則（10 秒 ~ 一次探索時間）。

擴建屬一次性大工程，可能落在較長時間段（數分鐘級），但**不超過「一次完整探索時間」的上限**。具體時長 TBD。

### 2.5 擴建敘事包裝

與委託系統的「交給我吧」敘事一致，擴建開始時村長夫人播放**擴建敘事對話**，結束時播放**完成對話**。

敘事對話內容由 `dialogue-writing-sop.md` 規範（TBD，可能屬於村長夫人的「工作中/主線承載」對話類別）。

## 三、擴建所需物資

### 3.1 所需物資 TBD

製作人 2026-04-17 第六輪 EA-4 決策：**擴建所需物資待後續 session 設定**。

### 3.2 設計原則（供後續 session 參考）

| 原則 | 說明 |
|---|---|
| **物資應為既有分類** | 優先使用 `type:素材`（與基地建設共用 pool），不新增分類 |
| **物資量應隨擴建次數遞增** | 第 1 次擴建較便宜、後續擴建成本漸高，提供有意義的長期目標 |
| **應強化探索價值** | 擴建物資應部分來自玩家親自探索（而非全部可透過守衛「探索周圍」取得），讓玩家感受「自己跑一趟」的必要性 |
| **可能整合禮物線/食物線物資** | 若需要食材或禮物類物資，將擴建作為物資出口之一，平衡經濟 |

具體物資清單與數量曲線由**數值設計師**後續補充。

### 3.3 物資檢查與扣除

- 擴建確認前，系統顯示所需物資與玩家目前持有量（合併背包 + 倉庫）
- 物資足夠 → [確認擴建] 按鈕亮起
- 物資不足 → 按鈕灰 + 提示缺少哪些物資
- 扣除順序：**先扣背包，背包不足時再扣倉庫**（與既有製作系統一致）

## 四、超出容量的處理

### 4.1 超出容量策略（TBD）

既有 GDD（random-ideas 第六輪 EA-6）列出三個選項：

| 選項 | 描述 |
|---|---|
| **A：完全拒絕放入** | 背包物品被迫留在背包，無法存入倉庫 |
| **B：強制玩家選擇丟棄/替換** | 彈出選單讓玩家決定捨棄哪些物品 |
| **C：溢位物品進入「臨時倉庫」** | 限時處置，逾期消失（複雜） |

製作人最終選擇為 TBD。本 GDD 暫以**選項 A**（完全拒絕放入）為工作假設，因其最簡潔且與格子制/堆疊制 UI 自然契合。

### 4.2 委託產出的溢位

委託領取時，若背包+倉庫都已滿：

- 參考選項 A：暫時無法領取，玩家需騰出空間
- 產出物資暫時保留於委託完成狀態（紅點持續亮）
- 與 `commission-system.md` 第六章 6.2「產出溢出處理」連動決策

## 五、與其他系統的接口

### 5.1 與村長夫人

- 擴建是村長夫人的主要交互循環之一（與主線任務、倉庫管理並列）
- 維持村長夫人接觸頻率：玩家每次想擴建都會進入她的 interact view
- 擴建敘事對話納入 G5 村長夫人內容量預算（與委託型角色的「交給我吧」對話不同分類）

### 5.2 與背包系統

- 倉庫規格與背包一致（格子制 + 堆疊），`StorageTransferManager` 轉移邏輯更簡化
- 物資扣除順序「先背包後倉庫」維持
- 背包容量本身的擴充機制（若有）與倉庫擴建獨立（TBD）

### 5.3 與委託系統

- 委託領取時若倉庫滿 → 依超出容量策略處理（與 5.1 統一）
- 擴建中的村長夫人狀態與委託中的角色工作中狀態一致（都使用「工作中 mm:ss」UI 模式）

### 5.4 與主線任務系統

- **倉庫擴建不依賴主線任務強推**（2026-04-20 設計轉向，委託教學流程移除後一併修正）
- 前期主線僅 T0「醒來的地方」、T1「認識所有人」、T2「出去看看外面」三條，皆不包含擴建引導
- 擴建為玩家自主發現的經營系統：玩家物資堆積 → 倉庫 UI 可見格子吃緊 → 自行前往村長夫人 [倉庫]→[擴建] 發起
- 若未來後期主線新增擴建相關事件（T5+），再於本節補充

### 5.5 與紅點系統

- 擴建完成觸發紅點屬 L1 層（委託完成層等級，詳見 `commission-system.md` 第四章）
- 與主線事件層（L4）可能並存，依紅點優先序處理（TBD）

### 5.6 既有程式碼影響

| 既有模組 | 改造方向 |
|---|---|
| `StorageManager` | 字典 → 格子陣列；新增 Capacity 與 Expand API |
| `StorageAreaView` | 顯示格子與堆疊、新增擴建入口 |
| `StorageTransferManager` | 格子對格子轉移邏輯 |
| `BackpackManager` | 結構已為格子+堆疊，邏輯可作為倉庫的參考實作 |

## 六、擴建的可見成長（核心樂趣 #2 驗證）

### 6.1 可見成長元素

| 元素 | 呈現方式 |
|---|---|
| **容量數值** | 倉庫 UI 顯示「已用格子 / 總格子」 |
| **空格子視覺** | 擴建前滿格密集、擴建後顯現大量空格 |
| **擴建里程碑** | 特定擴建次數可能觸發主線任務或 CG（TBD） |
| **擴建敘事回饋** | 村長夫人完成對話提供情感回報 |

### 6.2 長期驅動力

擴建是**持久性目標**：
- 每次擴建需要明確的物資投入（可見成本）
- 每次擴建帶來可見容量增長
- 與 Dark Room 的「再做一次」驅動力哲學一致
- 對應核心樂趣驗證標準 4「讓玩家想看下一步」
