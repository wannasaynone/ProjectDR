## 實作要點

### B2 DialogueManager 選項分支

設計決策：DialogueManager 不直接內嵌 VN 節點配置（例如不直接吃 NodeDialogueConfig）。原因：
- 保持 DialogueManager 為通用的對話引擎（純線性 + 可中途呈現選項 + 可中途延伸對話行）
- 讓節點劇情控制器（B9 開場劇情演出系統）作為呼叫端，負責：
  1. 從 NodeDialogueConfig 載入節點 → StartDialogue(intro lines)
  2. 推進至選項時機 → PresentChoices(node.Choices)
  3. 監聽 DialogueChoiceSelectedEvent → 依 ChoiceId 取得 node.GetResponseLines(branch) → AppendLines()
  4. 繼續 Advance() 至尾端發布 DialogueCompletedEvent

這樣設計的好處：
- DialogueManager 無需感知「節點」概念，感謝對話/一般對話/VN 對話通用同一介面
- 節點劇情控制器是薄的協調層，易於測試
- AppendLines 在對話已 complete 時可重新啟用，支援「選擇後延續」的流暢體驗

### B1 CharacterInteractionView 5 狀態

設計決策：五種狀態以 enum 統一管理，`SetState()` 統一入口。外部觸發：
- `Normal`：預設，既有行為完全保留
- `Forced`：返回按鈕隱藏，由節點 0 / 守衛歸來事件使用
- `FirstEntry`：邏輯上與 Normal 等同（返回按鈕可見），實際差異由呼叫端控制（播放登場 CG + 短劇情對話）
- `CommissionInProgress`：功能選單隱藏，倒數計時文字顯示（工作中 mm:ss）
- `CommissionReady`：功能選單隱藏，領取按鈕顯示，按鈕回呼由 `SetCommissionClaimCallback` 注入

VN 選項 UI 流程：
1. 接收 `DialogueChoicePresentedEvent` → 動態生成選項按鈕於 _choiceContainer
2. 對話區點擊被鎖（IsWaitingForChoice 期間不推進）
3. 選項按鈕點擊 → 呼叫 DialogueManager.SelectChoice → 清空容器 → 恢復對話區點擊 → AdvanceDialogue

### NodeDialogueConfigData 結構

對應 `node-dialogue-config.json`：
- JSON DTO `NodeDialogueConfigData { schema_version, note, node_dialogue_lines[] }`
- `NodeDialogueLineData { line_id, node_id, sequence, speaker, text, line_type, choice_branch }`
- 不可變 `NodeDialogueConfig` 依 node_id 分組、依 sequence 排序
- 每個節點產出 `NodeDialogueData { IntroLines, Choices, ResponseLinesByBranch }`
- `HasChoices` 便於呼叫端判斷節點是否為分支節點
