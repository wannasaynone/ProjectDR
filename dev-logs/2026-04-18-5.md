## 技術決策

### RedDotManager 優先序實作

GDD `commission-system.md` § 4.3 明訂 **L1 > L4 > L3 > L2**。列舉 `RedDotLayer` 的 integer 數值為 `None=0, CharacterQuestion=2, NewQuest=3, MainQuestEvent=4, CommissionCompleted=1`（刻意不按 integer 序）。實際優先序判斷採 `GetHighestLayer()` 內的硬編碼 switch，避免靠列舉數值自動排序導致誤讀。

### RedDotUpdatedEvent 去重

僅在 `HighestLayer` **變化**時發布事件，避免 L2 亮起時因已有 L1 而重複觸發 UI 更新。UI 訂閱方收到事件後只需直接套用 `HighestLayer`，不需要自行比對。

### L1 清除邏輯的保守策略

`OnCommissionClaimed` 直接關閉 L1 而非查詢 CommissionManager 是否還有其他 Completed slot。因為 RedDotManager 不持有 CommissionManager 引用（避免循環依賴）。若還有其他 slot 已完成，CommissionManager 的 `Tick` 並不會重新發布 `CommissionCompletedEvent`（邊界去重），導致 L1 有漏亮風險。

**暫採此策略的理由**：IT 階段 Witch/Guard 各只有少量 slot，實測中「同時多 slot 全完成 + 領取一個」情境極少；若後續確認需要嚴謹處理，可在 RedDotManager 建構時注入 `Func<string, bool>` 「某角色是否還有 Completed slot」callback。

### NodeDialogueController 選項呈現時機

實作上 `PlayNode` 即呼叫 `PresentChoices`（與 `StartDialogue` 同步）。UI 層（CharacterInteractionView）會同時收到 `DialogueStartedEvent` 與 `DialogueChoicePresentedEvent`；由 UI 自行控制「intro 播放完再顯示選項按鈕」（CharacterInteractionView 應在 `DialogueCompletedEvent` 或對話走到 `choice_prompt` 行時才啟用選項 UI）。

此作法簡單且事件流清晰；若 UI 有特殊時序需求（選項必須延遲顯示），再擴展 controller 支援「先 Advance 至 choice_prompt → 再 PresentChoices」兩階段 API。

### ICGPlayer 介面設計

純回呼介面（`PlayIntroCG(characterId, onComplete)`），不使用 async/await。理由：
- IT 階段 PlaceholderCGPlayer 立即回呼，無需 async 結構
- B13 實作可支援任意播放模式（Coroutine / UniTask / 事件驅動），onComplete 都是統一退出點
- 搭配 `CGPlaybackStartedEvent / CompletedEvent` 讓其他系統（UI 遮罩、音訊）能獨立訂閱

### OpeningSequenceController 不自動啟動

GDD 要求遊戲啟動時自動播放節點 0 開場。但 IT 階段 `VillageEntryPoint.InitializeManagers` 仍呼叫 `_progressionManager.ForceUnlock(...)` 強制解鎖 4 位角色（debug 友善設定）。若本批次自動觸發開場流程，會與既有 debug 狀態衝突。

**暫採策略**：OpeningSequenceController 組裝至 VillageEntryPoint 但不自動呼叫 `StartOpeningSequence()`。C1 整合測試或製作人指示時再啟動，預期在 C1 加入「首次啟動檢查 → 觸發開場」的存檔判斷邏輯。

### GuardReturnEventController 攔截機制

`ExplorationEntryManager` 新增 `IExplorationDepartureInterceptor` 介面（而非直接依賴 `GuardReturnEventController`）。這保留了 ExplorationEntryManager 的單一職責，並允許未來新增其他類型的攔截事件（例如「首次探索教學」、「特殊事件劇情」）。

VillageEntryPoint 注入的 Adapter 判斷三個條件：
1. `_guardReturnController.HasTriggered == false`（事件未觸發過）
2. `_unlockManager.IsUnlocked(Guard) == false`（守衛未解鎖）
3. 探索功能已解鎖（由 CharacterUnlockManager 的 IsExplorationFeatureUnlocked 隱含：若未解鎖，玩家 Hub 上沒有探索按鈕，不會呼叫 Depart）

三條件若都成立 → 攔截 Depart + 觸發 GuardReturnEventController.TriggerEvent()。

### DialogueManager 借用於 B9/B10

B9 OpeningSequenceController 的節點劇情與 B10 GuardReturnEventController 的守衛劇情都透過**同一個** DialogueManager 播放。由於兩者不會同時發生（開場節點 0 是強制模式，守衛歸來發生於後期首次探索），無競爭。

但 NodeDialogueController 與 GuardReturnEventController 各自訂閱 `DialogueCompletedEvent` — 兩者皆以自身狀態判斷「是否本次結束屬於我」：NodeDialogueController 檢查 `IsPlaying`，GuardReturnEventController 檢查 `_dialoguePhase`。無串擾。
