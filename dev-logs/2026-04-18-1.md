## 實作要點

### B6 StorageManager 容量化策略

**關鍵決策**：底層改為格子+堆疊，但保留舊字串 API 以保障既有相依模組（QuestManager、GiftManager、FarmManager、StorageTransferManager、StorageAreaView、GiftAreaView、FarmAreaView）不需改動。

- `GetAllItems()` 改為每次呼叫時由格子聚合為 `Dictionary<string,int>`（被既有 UI 層使用）
- `RemoveItem(itemId, int)` 仍維持 `bool` 回傳（從後往前掃格子，整批扣除）
- `AddItem` 在超容量時拋 `InvalidOperationException`（GDD 策略 A）；若呼叫端需要部分加入請改用 `TryAddItem`
- `DefaultInitialCapacity = 100`、`DefaultMaxStackValue = 99`（與 BackpackManager 一致）

既有 19 個 StorageManagerTests 的使用量均小（≤ 10），單格即可容納，測試通過。

### B6 擴建狀態機分離

`StorageExpansionManager` 作為獨立類別，依賴 `StorageManager` 與 `BackpackManager`：

- 狀態：Idle → InProgress → Completed → Idle（AcknowledgeCompletion）
- Tick(deltaTime) 推進倒數；歸零時自動 CompleteExpansion
- 物資扣除：HasEnoughResources 合併計算背包 + 倉庫；DeductResources 先背包後倉庫
- CompleteExpansion 呼叫 `StorageManager.ExpandCapacity(delta)` 實際擴建
- 事件：StorageExpansionStartedEvent / StorageExpansionCompletedEvent / StorageCapacityChangedEvent

### B3 CharacterUnlockManager 事件組裝

三個事件訂閱：
1. `DialogueChoiceSelectedEvent` → 節點 0 解鎖農女或魔女；節點 1 解鎖剩餘那位
   - 透過 `_node0ChosenBranch` 追蹤狀態。第一次 farm_girl/witch 觸發節點 0 邏輯；第二次且為另一位分支時觸發節點 1 邏輯
2. `MainQuestCompletedEvent` → T3 完成時解鎖探索功能（依 GDD 1.2 版：節點 2 結束後探索入口可見）
3. `GuardReturnEventCompletedEvent` → 解鎖守衛 + 派發 `guard_return_event` trigger 的 grant（贈劍）

初始狀態僅 `VillageChiefWife` 解鎖。建構時觸發 `node0_start` grant（依 initial-resources-config.json 節點 0 背包 placeholder 為空物品）。

`IInitialResourceDispatcher` 介面讓 CharacterUnlockManager 與 BackpackManager/StorageManager 解耦，方便測試（測試用 RecordingDispatcher 記錄派發列表）。

### B4 MainQuestManager 觸發模型

依據 JSON 中 `completion_condition_type` 欄位，管理器不自行偵測事件，而是提供兩個 API 給上層通知：

- `TryAutoCompleteFirstAutoQuest()` — T0 開局自動完成（type = "auto"）
- `NotifyCompletionSignal(signalType, signalValue)` — 其他類型由對應系統在適當時刻呼叫：
  - `dialogue_end` → 對話系統在特定對話完成時呼叫
  - `commission_count` → B5 CommissionManager 完成委託時呼叫
  - `first_explore` → B10 守衛歸來事件完成時呼叫（或首次探索管理器）
  - `first_storage_expand` → StorageExpansionManager 第一次完成時呼叫（目前 placeholder 未使用）
  - `first_char_intro_complete` → B13 登場 CG + 短劇情播放系統在首次完整播放完登場劇情時呼叫

這樣解耦設計讓 MainQuestManager 無需感知各子系統，只要各系統在對應時刻通知即可。

`unlock_on_complete` 欄位的解析：完成任務時會嘗試解鎖 pipe 分隔的每個項目 — 若目標是 Locked 狀態的任務 ID，則轉為 Available 並發布 `MainQuestAvailableEvent`；若是非任務 ID（如 "node_2_complete"、"exploration_open"），則由上層訂閱 `MainQuestCompletedEvent` 自行判讀處理。

### 新 DTO 全部遵循雙層模式

- JSON DTO 層：`[Serializable]` + snake_case 欄位
- 不可變 Config 層：pascal case 屬性 + 查詢 API（GetStage/GetQuest/GetGrant/GetGrantsByTrigger）
- 字串集合解析（pipe 分隔、key:value 分隔）在 Config 建構時一次完成
