## 根因總結：F7/F8/F9/F10 同一陷阱的連續發現

### 陷阱根源

守衛歸來事件（`GuardReturnEventController`）透過 `_cgPlayer.PlayIntroCG` 播放 CG，**繞過了 `InitializeCharacterView` 標準路徑**。`InitializeCharacterView` 的 PlayIntroCG callback 承載了多個 side effects，這些 side effects 在守衛歸來路徑中全都沒有被觸發。

### 連續 bug 發現順序

| Bug | 症狀 | 修復方式 | 問題 |
|-----|------|---------|------|
| F7 | DialogueManager 路徑無法完成 → 守衛不解鎖 | 廢棄 guard-return-config，CG 播完直接發布事件 | — |
| F8 | 玩家進入守衛 view 重播 CG | 在 `OnGuardReturnLockExploration` 補 `MarkIntroCGPlayed(Guard)` | 只補了一個 side effect |
| F9 | Hub 守衛按鈕 FirstMeet 紅點永不消失 | 在 `OnGuardReturnLockExploration` 補 `SetFirstMeetFlag(Guard, false)` | **方向錯誤**（太早清除，違反 FirstMeet 語義） |
| F10 | Hub 沒紅點 + 進入守衛 view 沒觸發對話 | 系統性重構（本次） | — |

### F9 為何方向錯誤

`RedDotManager.SetFirstMeetFlag` 是「玩家首次點擊進入該角色互動畫面」時的清除信號，代表玩家已完成首次登場 CG，不再需要紅點引導。

F9 把這個清除放在守衛歸來完成瞬間 → Hub 按鈕出現時就沒有紅點 → 玩家看不到引導 → 不知道要去找守衛發問取劍 → 探索永遠無法解鎖。

## Phase 1：Callback side effects 盤點

`InitializeCharacterView` 中 PlayIntroCG callback 的 side effects：

| 分類 | 程式碼 | 說明 |
|------|--------|------|
| **(a) CG 播放本身** | `MarkIntroCGPlayed(capturedCharId)` | only-once 旗標，防止重播 |
| **(a) CG 播放本身** | `capturedView.SetState(Normal)` | CG 完成後切回 Normal 狀態 |
| **(b) 首次登場狀態** | `_redDotManager.SetFirstMeetFlag(capturedCharId, false)` | 玩家已進入，清除 Hub 引導紅點 |
| **(c) 常規進入流程** | `SetMainQuestEventFlag(VCW, true)` | 設定 VCW L4 紅點，引導回村長夫人 |
| **(c) 常規進入流程** | `_node1TriggerReady = true` 或 `_node2TriggerReady = true` | 觸發主線節點，依節點 1 是否已播決定 |

備注：(c) 類只對農女/魔女觸發（`capturedCharId != VCW && capturedCharId != Guard`）。守衛不觸發節點 1/2。

## 新路徑設計

### 抽出共用方法 `OnCharacterEnteredAndCGDone`

**位置**：`VillageEntryPoint.cs`，緊鄰 `InitializeCharacterView`

**職責**：執行 (b)/(c) 類 side effects，供兩條路徑共用：

```
路徑 A（標準）：
  HasPlayedIntroCG = false → 播 CG → callback
    → MarkIntroCGPlayed
    → SetState(Normal)
    → OnCharacterEnteredAndCGDone ← 執行 (b)/(c)

路徑 B（守衛特例）：
  HasPlayedIntroCG = true → 跳過 CG
    → OnCharacterEnteredAndCGDone ← 執行 (b)/(c)
```

**方法內容**：
- (b) `SetFirstMeetFlag(characterId, false)` — 清除 Hub 引導紅點
- (c) 若 `characterId != VCW && characterId != Guard`：
  - `SetMainQuestEventFlag(VCW, true)`
  - 依 `_playedMainQuestNodes` 設定 `_node1TriggerReady` 或 `_node2TriggerReady`

## 如何避免未來同類 bug（MEMORY 教訓）

**陷阱名稱**：繞道路徑系統性補償遺漏

**陷阱描述**：  
當某個事件（如守衛歸來）透過旁路（`_cgPlayer.PlayIntroCG`）觸發了原本只在標準路徑（`InitializeCharacterView` callback）執行的 side effects 時，修復者傾向於「單點補救」——在新路徑的 handler 中一個一個補上缺少的副作用。

這種方式的問題是：
1. 每次修復只看到眼前的症狀，容易遺漏其他 side effects
2. 修復後的程式碼散落在多個地方，難以追蹤
3. 語義違反（如 FirstMeet 清除時機）很難被察覺

**正確做法**：  
遇到「繞道路徑」時，應系統性分析原始路徑的所有 side effects，並抽出共用方法，讓所有路徑都呼叫同一個入口，而不是每條路徑各自補齊。

**決策依據**：Sprint 6 F10 系統性重構，2026-04-20。
