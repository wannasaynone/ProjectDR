# Dev Log 2026-04-20-11

**任務**：Sprint 6 F9 bugfix（D4 bugfix 6）— 守衛 Hub 按鈕 FirstMeet 紅點永不消失

**關聯 Sprint 項目**：F9（sprint-6-explore-gate-rework.md）

---

## 根因分析

### F8 修復的副作用

F8（2026-04-20-10）修復了兩個 bug，其中 Bug 1 的修復在 `OnGuardReturnLockExploration` 中補呼叫 `MarkIntroCGPlayed(Guard)`，讓守衛歸來 CG 不被重播。

但 `InitializeCharacterView` 的 CG callback 除了 `MarkIntroCGPlayed` 外，還有另一個副作用：

```csharp
// InitializeCharacterView 中 CG callback 的原始程式碼
_redDotManager.SetFirstMeetFlag(capturedCharId, false);
```

F8 修復讓 Guard 的 CG block 被跳過，導致 `SetFirstMeetFlag(Guard, false)` 從未被呼叫。

### Bug 表現

守衛解鎖後（`CharacterUnlockedEvent` 觸發 `SetFirstMeetFlag(Guard, true)`），Hub 按鈕出現 FirstMeet 紅點。

按照設計：玩家進入守衛互動畫面後，CG 播完，FirstMeet 旗標應被清除。

但 F8 後，玩家從守衛歸來 CG → Hub 按鈕 → 點守衛按鈕，FirstMeet 旗標永遠不清除（因為沒有 CG callback 呼叫 `SetFirstMeetFlag(false)`）。

---

## 修復方案

**最小修復**：在 `VillageEntryPoint.OnGuardReturnLockExploration()` 中補呼叫 `_redDotManager.SetFirstMeetFlag(Guard, false)`，與 F8 的 `MarkIntroCGPlayed` 一起完成 CG callback 的兩個副作用。

## 技術決策

1. **選擇在 `OnGuardReturnLockExploration` 中修復**：此 handler 已是守衛歸來 CG 完成後的「集中補償點」，F8 已在此補 `MarkIntroCGPlayed`，F9 的補救也應在此進行，保持修復的一致性與可讀性。

2. **不修改 `InitializeCharacterView`**：F8 的修復已讓 CG block 被跳過，在 `InitializeCharacterView` 中加 else-case 可能會引入複雜的條件，不如直接在事件 handler 中補上缺少的副作用。
