# 角色頁面「對話」功能流程修正

> **建立日期**: 2026-04-18
> **狀態**: ✅ 已定案（2026-04-18）
> **相關主題**: `gdd/character-interaction.md` v2.2、`gdd/character-content-template.md` v1.0、`gdd/commission-system.md` v1.1

## 內容

製作人 2026-04-18 指出先前對角色頁面「對話」按鈕的理解有誤，正確流程如下：

### a. 角色發起對話（紅點觸發路徑）

1. 村莊（Hub）該角色頭像顯示紅點
2. 玩家進入該角色頁面，頁面內「對話」按鈕顯示紅點
3. 玩家點擊「對話」 → **關閉右下選單** → 進行對話
4. 對話結束 → 右下選單出現**四個選項**，分別為 **+0 / +2 / +5 / +10**（暫定測試數值）
5. 玩家選擇其中一個 → 播放對應對話
6. 對話完畢 → **回復原本的功能選單**（四項：專業/任務/對話/送禮）

### b. 玩家主動發起對話（無紅點路徑）

1. 玩家點擊「對話」 → **關閉右下選單**
2. 顯示**四個問題**給玩家選
3. 玩家選擇後 → 角色給予對應的回答
4. 回答完畢 → **回復原本的功能選單**

## 補充與演變

### 2026-04-18 第二輪（製作人補充）

製作人明確三個先前未處理事項：

2. **體力機制保留** — §2 角色體力條與 §3 對話消耗體力的設計維持
3. **回報類型 = 進入自動播放** — L1 完成回報、L4 主線事件類紅點進入角色頁面後**自動播放**對話（與 `character-interaction.md` v2.2 §5.1 既有規格一致）

### 新流程分類釐清

根據以上回答，紅點機制現在分為兩種行為模式：

| 紅點類型 | 進入行為 | 對應 UX |
|---|---|---|
| **L1 委託完成** | 進入角色 → **自動播放**完成回報對話 | 既有規格 §1.3 |
| **L4 主線事件** | 進入角色 → **自動播放**事件對話 | 既有規格 §1.3 |
| **L2 角色發問**（對話類） | 進入角色 → 「對話」按鈕**紅點** → 玩家**點擊才觸發** | 🆕 新流程 a |
| **L3 新任務** | 進入角色 → `[任務]` 按鈕**紅點** → 玩家點擊才觸發 | 既有規格（本來就是手動） |

→ 原矛盾 #1「自動 vs 手動」**化解**：不是全面取代，是**按紅點類型分流**。
→ 矛盾 #2/#3 仍需處理（角色發問觸發條件、玩家發問切換條件）。

## 待確認問題

### ✅ 已解答（2026-04-18 第二輪）

- **Q1**：紅點觸發改為手動是否全面取代？→ **否**，按紅點類型分流（L1/L4 仍自動播放，L2/L3 為按鈕紅點手動點擊）

- **Q7**：「對話」按鈕紅點是否只適用 L2？→ **是**，L1/L4 維持 §1.3 既有完成領取狀態流程

### ✅ 第三輪全部解答（2026-04-18 製作人回覆）

- **Q2** — 玩家發問：**抽四題**（維持 40 題池每次抽 4 題顯示，與 v1.0 §3.3 機制一致）
- **Q3** — 角色發問：**維持 70 題**
  - 紅點觸發規則：**等待固定時間**倒數 → 時間到出紅點
  - **角色工作中時不倒數**（角色正在執行委託 = 倒數暫停）
- **Q4** — **就這四種**（+0 / +2 / +5 / +10 就是四個個性類型對應的固定數值，無偏好/次偏好/中性/錯誤之分，而是每個個性對應一檔）
- **Q8** — 體力消耗：**玩家主動發話時消耗**（紅點觸發 a 路徑不消耗；玩家主動 b 路徑消耗）
- **Q9** — 招呼語 vs 紅點：**取代**（紅點亮時，紅點對話取代招呼語；無紅點時才播招呼語）
- **Q10** — 玩家發問回答：
  - 預設每題**固定 1 對 1** 回答
  - 當 40 題問題池**全部用完**後，出現 **[閒聊]** 選項
  - 選 [閒聊] → 角色隨機發問 + 隨機回答（原 v1.0 §3.3「3 種隨機回答」機制被重新定位為閒聊模式內容）

### 新增規則（第三輪新增）

- **玩家主動對話冷卻時間**：玩家點「對話」後有冷卻時間（CD 數值 TBD）
- **工作時冷卻加倍**：角色正在執行委託時，玩家主動對話的冷卻時間 × 2

### 第三輪新增張力的製作人答覆（2026-04-18 第四輪）

**張力 A — 工作中不能點對話（定案）**
- §1.2 規格**維持不變**：工作中無功能按鈕可點，僅能退出
- 「玩家主動對話冷卻」在工作中仍在**計算規則中運作**（時間×2 加倍扣減），但不顯示
- 即使冷卻時間在工作中到了，也不會顯示（因為整個選單都被倒數取代）
- 設計意圖：冷卻機制是**計算規則**，不是 UI 元素

**張力 B — 工作中紅點延後顯示（定案）**
- 工作中選單被倒數取代，對話按鈕不可見，紅點無法顯示
- 若紅點在工作前或工作中累積 → 工作完成領取後才能看到
- 具體行為：**當下對話就會亮紅點**（玩家完成領取、回到正常選單狀態的當下，若倒數已到，紅點會立即亮起）

**張力 C — L1/L2 同時累積的流程（定案）**
- 流程：進入 → 自動播 L1 完成回報 → 結束 → 對話按鈕保留 L2 紅點 → 玩家點擊觸發
- 需在 GDD 明寫「L1/L4 自動播放結束後，若 L2/L3 紅點仍存在則保留顯示」

**張力 D — CD 數值由數值設計師填寫（定案）**
- 撰寫 GDD 時將 CD 標記為 TBD + 委派數值設計師填寫一版初始數值
- 工作時 ×2 倍率為規則層定案，數值設計師不能改此倍率

**張力 E — 閒聊選項規則（定案）**
- **池耗盡（0 題剩餘）**：只顯示一個 [閒聊] 選項
- **有剩餘題目（1~4+ 題）**：顯示所有剩餘題目，不顯示閒聊
- **剩餘 1~3 題**：只顯示該 1~3 題（不補 [閒聊]、不補滿 4 格）
- **剩餘 ≥ 4 題**：顯示 4 題（選題邏輯 TBD — 隨機抽 4 / 固定順序？）

### 第五輪（2026-04-18）製作人補充數值

1. **剩餘題目 > 4 時的選題邏輯**：**隨機抽 4 題**
2. **玩家主動對話冷卻 CD 基礎值**：**60 秒**（工作中 ×2 = 120 秒）
3. **角色發問倒數時間基礎值**：**60 秒**

以上為 placeholder 初始值，數值設計師後續可調整。規則層（工作中 ×2 倍率、倒數工作中暫停）不變。
