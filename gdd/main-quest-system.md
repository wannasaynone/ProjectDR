# 主線任務系統（Main Quest System）

> **版本**: v2.4

## 系統目的

主線任務系統承接 `core-definition.md` v3.1 的核心樂趣驗證標準 4「是否創造讓玩家想看下一步的驅動力？」，同時是本作對 A Dark Room **漸進揭露哲學**的落地實現。

本系統的定位明確為**機制引導（Mechanics Onboarding）**（製作人 2026-04-17 第十輪 JA 決策）：

- **任務不是**史詩劇情
- **任務不是**角色支線
- **任務不是**日常隨機內容
- **任務 = 機制引導** — 每個任務是一個機制／功能的敘事包裝，其存在是為了告訴玩家「你可以做 X」

> **2026-04-20 設計轉向**：委託教學已從前期主線序列中移除（決策 1/2）。前期任務序列更精簡為 T0/T1/T2 三條，委託系統改為玩家自主發現。T2 開始後的中後期任務範式不受影響。

對應 Dark Room 的漸進揭露節奏（stoke fire → gather wood → check traps → hunt → 村民到來 → 戰鬥 → 太空艙），ProjectDR 的主線任務序列是「認識所有角色（T1）→ 首次探索（T2）→ 倉庫擴建 → 深探新區域 → 新配方解鎖 → ……」。

**任務完成 ≠ 破關**（製作人 2026-04-17 第五輪 DB-b 決策，與 4/16「無終局」一致）：主線任務只推進機制揭露與系統解鎖，玩家在完成所有既有主線任務後仍可無限遊玩。

## 一、任務的本質：機制引導

### 1.1 任務設計範式

每個主線任務 = 一個**機制／功能的敘事包裝**。

中後期任務範式由製作人後續分批決定，不在此列舉範例。

> 前期（角色解鎖階段）的 T0~T2 主線任務序列已定案，見下方 1.4 章節。

### 1.2 設計優勢

1. **工作量可控**：任務數量不固定，可視製作節奏分批新增
2. **新手教學自然化**：不需獨立教學關卡，主線就是引導
3. **無內容壓力**：任務 ≠ 劇情主菜，可以是短序列對話
4. **增量擴充友善**：新機制加入遊戲時，只需加一段主線任務作為引導
5. **Dark Room 節奏哲學落地**：漸進揭露 = 主線任務序列

### 1.3 任務與其他系統的敘事分工

| 系統 | 敘事功能 |
|---|---|
| **主線任務** | 機制引導（告訴玩家可以做什麼、下一步在哪） |
| **角色對話（G5 招呼/發問/玩家發問）** | 角色個性展示、戀愛養成情感累積 |
| **里程碑對話 / 約會 / H 劇情** | 好感度進展的情感回報 |
| **委託的「交給我吧」與完成對話** | 角色在日常勞動中的形象 |

主線任務不與角色支線、日常任務競爭——它是**系統層的引導**，角色層的敘事由 G5 的對話系統承載。

### 1.4 前期主線任務序列（T0~T2）

> **2026-04-20 設計轉向（決策 5）**：原 T1/T2/T3 合併為新 T1「認識所有人」；原 T4 重編為新 T2「出去看看外面」。委託教學（原 T2/T3 完成條件）完全移除，不再作為任何前期主線任務的觸發或完成條件。

**前期主線任務同時扮演「角色解鎖觸發器」的角色**——每個節點的觸發條件直接由角色解鎖進度或事件觸發（`character-unlock-system.md` v1.3）。

前期主線任務序列（重編後）如下：

| 任務 | 引導的機制 | 觸發節點 | 完成條件 |
|---|---|---|---|
| **T0 開局** | 認識村長夫人、世界觀介紹 | **節點 0**（開局自動） | 開局自動完成（節點 0 對話結束） |
| **T1「認識所有人」** | 角色解鎖循環（VN 式選項 → 登場 CG → 節點 1/2 劇情） | **節點 1**（選擇 1 角色 CG 完成）→ **節點 2**（魔女對話結束，第二位完成，無論先後） | 節點 2 劇情完成（村長夫人三次對話全部完成）|
| **T2「出去看看外面」** | 首次探索 + 守衛歸來事件 + 首次進入守衛 interact view 取劍對白完成 | **守衛歸來**（首次探索自動觸發） | 首次探索完成 + 守衛歸來事件完成 + **首次進入守衛 interact view 取劍對白結束**（T2 完成 = 劍入手 + 探索重開） |

**觸發流程補充說明**：

- T1 開始：節點 0 完成後自動觸發
- T1 完成：節點 2 劇情完成（村長夫人第三次對話結束）→ 探索入口解鎖
- T2 開始：探索入口解鎖後（T1 完成後可進入探索）
- T2 完成：首次探索 + 守衛歸來事件完成 + 首次進入守衛 interact view 取劍對白結束 → 劍入手 + 探索入口重開（守衛 Hub 按鈕在守衛歸來事件完成時即解鎖；取劍對白在玩家首次進入守衛 interact view 時自動觸發）

> **重要關聯**：T0~T2 的完成狀態直接驅動 `character-unlock-system.md` v1.3 的節點觸發。任務層與解鎖層是**同一條時間軸的雙重視角**——從任務角度看是「機制引導序列」，從解鎖角度看是「角色加入節奏」。

### 1.5 前期任務與後期任務的關係

- **前期**（T0~T2）：任務 = 解鎖觸發器 + 機制引導（雙重角色）。前期只有 3 條任務，精簡設計讓玩家快速進入完整循環。
- **後期**（T3 起）：任務 = 純粹機制引導（解鎖已全數完成，任務不再承擔解鎖責任）

前期任務的具體對話字數、完成條件細節、獎勵配置為 TBD，由 `character-unlock-system.md` v1.3 實作階段定案。

## 二、任務結構

### 2.1 資料結構（概念）

```
MainQuestData {
    QuestId              // 任務唯一識別
    Title                // 任務標題
    Description          // 任務描述（展示在承接提示視窗）
    DialogueLines        // 任務承接時的對話（在 interact view 播放）
    TriggerConditions    // 觸發條件清單（詳見 2.2）
    CompletionConditions // 完成條件清單（詳見 2.3）
    OwnerCharacterId     // 任務擁有者（哪個角色的 [任務] 按鈕亮紅點）
    Rewards              // 獎勵（詳見 2.4）
    UnlocksNextQuests    // 觸發後續哪些任務（事件序列）
}
```

### 2.2 觸發方式：事件序列 + 觸發條件

製作人 2026-04-17 第五輪 DB-a 決策：任務採**事件序列 + 觸發條件**結構。

觸發條件可為下列維度的組合（支援 AND／OR 組合，TBD）：

| 觸發條件 | 範例 |
|---|---|
| **前置任務完成** | 前置任務完成後解鎖後續任務（如新 T0 完成後解鎖新 T1） |
| **村莊等級** | 探索/委託進度累計達門檻（若保留此概念，TBD） |
| **好感度** | 特定角色好感度達門檻 |
| **物資收集** | 特定稀有物資出現於倉庫／背包 |
| **時間／遊戲日數** | 開局 X 分鐘後或 X 日後自動 |
| **系統行為** | 倉庫使用率達 70%、首次使用某功能等 |

事件序列意味著：任務之間有**明確的順序與依賴關係**，而非全部並行開放。

### 2.3 完成條件

完成條件（TBD，依每任務性質而定）：

| 條件類型 | 範例 |
|---|---|
| 對話完成 | 承接後進入指定角色對話序列結束 |
| 委託行為 | 完成一次耕種／煉製／探索周圍（與委託系統整合） |
| 探索行為 | 抵達特定地圖／格子／擊敗特定魔物 |
| 經營行為 | 倉庫擴建一次／送禮一次／好感度達 X 階段 |
| 混合 | 上述多項達成 |

### 2.4 獎勵類型

| 獎勵類型 | 說明 |
|---|---|
| **新機制解鎖** | 新委託類型、新配方、新區域 |
| **新地圖／探索區域解鎖** | 與探索系統整合 |
| **角色 CG／劇情** | 與既有 CG 解鎖系統整合 |
| **倉庫容量擴展加成** | 與擴建系統整合（可能作為擴建的替代路徑之一，TBD） |
| **世界觀揭露** | Dark Room 式的文字揭露，但限於戀愛養成所以不走外星人式翻盤敘事 |
| **物資獎勵** | 任務完成給予一次性物資 |

## 三、任務按鈕與承接流程

### 3.1 任務按鈕位置

**Sprint 6 設計轉向後（2026-04-20 實作對齊）**：T0/T1/T2 皆自動完成或由對話觸發，不需玩家點 [任務] 按鈕承接。功能選單改為 `[主功能][送禮][CG 圖鑑][對話]`：

| 角色 | 功能選單 |
|---|---|
| 村長夫人 | `[倉庫][送禮][CG 圖鑑][對話]` |
| 農女 | `[耕種][送禮][CG 圖鑑][對話]` |
| 魔女 | `[煉製][送禮][CG 圖鑑][對話]` |
| 守衛 | `[探索周圍][送禮][CG 圖鑑][對話]` |

> Sprint 6 設計轉向後，T0/T1/T2 皆自動完成或對話觸發，不需玩家點 [任務] 按鈕承接。

### 3.2 任務按鈕的紅點顯示

**T0~T2 自動觸發，無 [任務] 按鈕紅點。**

T0~T2 的推進由事件層（L4）與對話系統自動處理，不存在需要玩家主動承接的任務按鈕，因此 L3（任務層）紅點在前期序列中不啟用。

未來若加入 T3+ 中後期任務並採用「角色承接任務」模式，才會恢復此紅點層設計。屆時規則如下（保留供未來 T3+ 擴充參考）：

| 狀態 | 紅點顯示 |
|---|---|
| 該角色有可承接任務 | `[任務]` 按鈕顯示紅點 |
| 該角色無可承接任務 | 無紅點 |
| 玩家已承接 | 紅點消失 |
| 完成後需回報（TBD，詳見 JA-3） | 可能再次亮紅點 |

紅點層級關係詳見 `commission-system.md` 第四章「紅點觸發四層」：L3 為任務層，L4 為事件層。

### 3.3 承接流程

**T0~T2 無承接流程（自動完成）。**

T0~T2 的任務均由系統自動推進：

- **T0**：開局節點 0 對話結束後自動完成
- **T1**：節點 2 劇情完成（村長夫人第三次對話結束）後自動完成
- **T2**：首次探索完成 + 守衛歸來事件完成後自動完成

未來若加入需玩家主動承接的任務類型（T3+），將於該 Sprint 補此章節。屆時承接流程骨架如下（保留供未來 T3+ 擴充參考）：

```
玩家進入角色 → 對話播放 → 功能選單 [XX][任務][對話][送禮]
  ↓ 看到 [任務] 按鈕紅點
玩家點擊 [任務]
  ↓
Interact View 中播放對話（任務描述對話）
  角色說出任務背景與引導
  ↓ 對話結束
跳出「承接任務」提示視窗
  - 任務標題
  - 任務描述
  - 完成條件（如適用）
  - [接受] / [拒絕] 按鈕
  ↓ 玩家選 [接受]
任務進入「進行中」狀態
  ↓ 返回 interact view（或 Hub）
```

### 3.4 拒絕的處理（TBD）

玩家拒絕承接後的行為：

- 選項 C：任務標記為「已拒絕」，不可再承接（極少見，TBD）

## 六、主線完成 ≠ 破關

### 6.1 設計決策（DB-b）

製作人 2026-04-17 第五輪 DB-b 決策：**主線任務全部完成不代表遊戲結束**。

- 與 `core-definition.md` 決策 #5「無終局」一致
- 完成所有主線任務後，玩家仍可：
  - 繼續進行各角色的戀愛養成（好感度、送禮、CG）
  - 繼續執行各項委託與探索
  - 享受已解鎖的全部系統內容
- 主線任務的終點是「所有機制都已被引導認識」，而非「遊戲結束」

### 6.2 主線任務的盡頭敘事

當最後一段主線任務完成時，適合的敘事呈現（TBD）：

- 短段世界觀揭露（輕度 Dark Room 式）
- 村莊狀態的某種「新常態」描繪
- 不走翻盤敘事（不是太空艙離開，不是外星人揭密）
- 保持戀愛養成基調

## 八、與其他系統的接口

### 8.1 與委託系統

- **主線不再依賴委託完成**（2026-04-20 設計轉向，決策 1/2）：前期主線 T0~T2 的觸發與完成條件中不含任何「完成一次委託」要求
- 委託系統為玩家自主發現的次要系統；玩家可能玩很久都不碰委託，也可能第一次就發現（進入任意角色 interact view → 自由點選功能選單）
- 中後期任務仍可能引導特定委託（作為機制深度引導），但不作為解鎖前置
- 任務獎勵可能解鎖新配方（擴展各 Manager 的配方表）
- 詳見 `commission-system.md` v1.3 第一章「非強制定位」

### 8.2 與倉庫擴建系統

- 中後期任務可能引導擴建機制（後續由製作人決定）
- 任務獎勵可能提供額外容量（TBD）
- 詳見 `storage-expansion.md`

### 8.3 與探索系統

- 任務可能引導玩家進入特定地圖／深探特定格子
- 任務獎勵可能解鎖新地圖區域
- 與 `exploration-system.md` 既有 55 條規則整合

### 8.4 與好感度／CG 系統

- 任務獎勵可能包含 CG 解鎖（既有 `CGUnlockManager`）
- 任務觸發條件可能依賴好感度門檻

### 8.5 與紅點系統

- 任務紅點屬於紅點系統的 L3（任務按鈕紅點）與 L4（事件層主線對話）層
- 詳見 `commission-system.md` 第四章「紅點觸發四層」

### 8.6 新增程式模組（TBD，實作階段處理）

- `MainQuestManager`（任務狀態管理、觸發條件判定、完成判定）
- `MainQuestData`（任務資料結構）
- 任務承接提示視窗 UI（`QuestAcceptView` 或類似）
- 任務清單 View（若採全域任務面板方案）
