## 根因分析

### Sprint 4 原始路徑（已被 C5 刪除）

Sprint 4 C2（`dev-logs/2026-04-18-7.md`）的 `OnNodeDialogueCompletedForMainQuest` 在 node_1 完成時呼叫：
1. `CharacterUnlockManager.ForceUnlock`（解鎖剩下那位）
2. `DispatchInitialResourceGrants(UnlockFarmGirl/UnlockWitch)`（發放物資）

### C5 R3 刪除了什麼（`dev-logs/2026-04-20-3.md` R3）

R3 移除：
- `CharacterUnlockManager.ForceUnlock` 呼叫
- `DispatchInitialResourceGrants(UnlockFarmGirl/UnlockWitch)` 呼叫
- 整個 `DispatchInitialResourceGrants(string triggerId)` private 方法

R3 備註說明：「改由 `OnDialogueChoiceSelected` 在節點 1 VN 選項確認時處理」

### 但 OnDialogueChoiceSelected 路徑從未能正常運作

關鍵問題：`OnDialogueChoiceSelected` 只處理 `branch == NodeDialogueBranchIds.FarmGirl || branch == NodeDialogueBranchIds.Witch` 的情況。

但真實 `node-dialogue-config.json` 中節點 1 的選項：
```json
{
  "line_id": "node1_choice_meet",
  "node_id": "node_1",
  "sequence": 6,
  "speaker": "player",
  "text": "好，帶我去見她。",
  "line_type": "choice_option",
  "choice_branch": ""    // ← 空字串！
}
```

空字串的 choice_branch 不匹配任何 branch，`OnDialogueChoiceSelected` 永遠不會觸發解鎖。

### 為什麼 Sprint 4 測試沒有發現這個問題？

`NodeProgressionIntegrationTest.BuildNodeDialogueConfig()` 的節點 1 選項使用了 `farm_girl`/`witch` 兩個分 branch：
```csharp
// 測試 config（理想化，與真實 config 不符）
new NodeDialogueLineData { line_id = "n1_c1", ..., choice_branch = "farm_girl" },
new NodeDialogueLineData { line_id = "n1_c2", ..., choice_branch = "witch" },
```

測試通過，但 production config 的節點 1 只有一個空 branch 選項。測試 config 與 production config 不一致，導致 bug 在測試層被遮蔽。

### 根因的時序鏈

1. Sprint 4 C2：原始路徑 = `OnNodeDialogueCompletedForMainQuest` node_1 分支直接 `ForceUnlock`（有效）
2. C5 R3：刪除此路徑，改依賴 `OnDialogueChoiceSelected`（理論上有效，但 config 端沒有對應 branch）
3. Config（`node-dialogue-config.json`）節點 1 選項 branch 一直都是空字串 → 解鎖路徑從未被建立
4. Sprint 4 測試用了不同於 production 的 config → 未發現

**核心教訓**：測試用 config 與 production config 的節點 1 選項結構不一致，導致 bug 在測試層被遮蔽。未來測試 config 應盡量對齊 production config 的真實結構，尤其是 choice_branch 這類影響業務邏輯的欄位。

## 修復方案

採用**方案 A（最小改動）**：在 `CharacterUnlockManager` 新增訂閱 `NodeDialogueCompletedEvent`，node_1 完成時依 `_node0ChosenBranch` 推算剩下那位並 `ForceUnlock`。

選擇在 `CharacterUnlockManager`（而非 `VillageEntryPoint`）實作的理由：
1. `CharacterUnlockManager` 已是純邏輯 IDisposable，可在 Editor 測試中直接驗證
2. `VillageEntryPoint` 是 MonoBehaviour，無法在純邏輯測試中驗證其 `OnNodeDialogueCompletedForMainQuest`
3. 解鎖狀態本來就是 `CharacterUnlockManager` 的職責範圍

**不採用方案 B**（VN 選項 config 新增 branch_id）：需改 config，屬於設計決策範疇，須先回報製作人。且真實場景節點 1 的選項語義就是「無條件確認」，強加 branch 反而不自然。
